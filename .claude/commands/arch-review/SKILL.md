# Arch Review

シニアエンジニアの視点でコードベースを分析し、問題をGitHubのIssueとして作成し、分析結果を`.analysis/`に保存します。

## 実行手順

### Step 1: 対象の特定

引数 `$ARGUMENTS` を確認してください：

- **引数なし** → `src/` 全体を対象に分析
- **ディレクトリ・ファイルパス** → 指定された範囲を対象に分析
- **機能名・キーワード** → Glob/Grepで関連ファイルを特定してから分析

### Step 2: コードの読み込みと分析

対象コードを読み込み、以下の観点で問題を洗い出してください。**表面的なコードレビューではなく、設計の根拠・パターンが存在する理由まで掘り下げること。**

**設計の正確性**

- DDDの概念（エンティティ・値オブジェクト・ドメインサービス）の正しい適用
- IDの安定性（`randomUUID()`の誤用がないか）
- 責務の分離が実際に機能しているか

**型安全性**

- `as`キャストによる型チェックの回避
- `never`型を使った網羅性チェックの欠如
- `unknown`/`any`が適切に扱われているか

**Reactパターン**

- `useSyncExternalStore`の参照安定性
- モジュールレベルのミュータブル状態
- key属性の安定性

**非同期処理**

- `Promise.allSettled`の失敗が無音でないか
- AbortControllerによるリクエストキャンセル
- Service Workerのライフサイクル対応

**アーキテクチャ整合性**

- `.claude/rules/service-architecture.md` のアクセスルールとの乖離
- DIが実際にテスタビリティに寄与しているか
- キャッシュ無効化戦略

**オブザーバビリティ**

- エラーの握り潰し（グレースフルデグラデーションと区別する）
- デバッグ時に原因を追えるか

### Step 3: 問題のトリアージ

見つかった問題を分類してください：

- **重大**: 設計の根本が間違っている・将来の機能開発を阻害する
- **中**: 特定条件で顕在化する・技術的負債として蓄積する
- **低**: デバッグ容易性・拡張性の改善

**問題が0件の場合**: Issue・ファイルは作成せず、「問題は見つかりませんでした」と報告して終了。

### Step 4: GitHubのIssueを作成

問題ごとに1つのIssueを `gh issue create` で作成してください。

**タイトル形式**: `fix:` / `feat:` / `refactor:` + 問題の要旨（英語）

**本文の構成**:

```
## 問題
[何が問題か・なぜ問題かを具体的に。現状コードのスニペットを含める]

## 影響範囲
[影響するファイル・関数・ユーザー体験]

## 改善案
[具体的な修正方法をコードスニペット付きで。複数案はOption A/Bで列挙]

## 不足している知識（任意）
[このバグ・設計ミスの背景にある概念を説明する]

## 優先度
**重大/中/低** — [一言で理由]
```

**ラベルの付与**: `gh issue edit` で以下を付与してください：

- 優先度: `priority: high` / `priority: medium` / `priority: low`
- 種別: `bug` / `enhancement` / `refactor` から該当するもの
- 場所: `backend` / `service` / `hooks` / `components` から該当するもの

### Step 5: 分析結果をファイルに保存

`.analysis/YYYY-MM-DD-<対象名>-review.md` に分析結果を保存してください。日付は実行日を使用。

**ファイルの構成**:

```markdown
# <対象名> レビュー

**日付**: YYYY-MM-DD
**対象**: <分析対象のパス>
**評価**: <総合評価と一言コメント>

---

## 問題一覧

| #   | 問題 | 深刻度 | 関連Issue |
| --- | ---- | ------ | --------- |

---

## 問題詳細

### 問題N: <タイトル>【深刻度】

**該当ファイル**: ...
**現状**: コードスニペット
**何が起きているか**: 説明
**改善案**: コードスニペット
**不足している知識**: 概念の説明

---

## 知識ギャップのまとめ

| テーマ | 現状の理解レベル | 学習すべき内容 |
| ------ | ---------------- | -------------- |

---

## 総評

[設計の方向性・優先対応順・全体所感]
```

### Step 6: 結果のサマリー

以下の形式でユーザーに報告してください：

```
## 作成したIssue

| Issue | タイトル | 深刻度 |
|---|---|---|
| [#XXX](URL) | タイトル | 重大/中/低 |

## 分析ファイル

`.analysis/YYYY-MM-DD-<対象名>-review.md` に保存しました。
```

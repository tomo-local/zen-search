# Arch Analyze

コードを読み込んで設計上の問題を洗い出し、`.analysis/` に保存します。
IssueやPRは直接作成しません。問題の対応方針を検討したい場合は `/arch-plan`、Issueを作成したい場合は `/arch-issue` を使ってください。

## 実行手順

### Step 1: 対象の特定とスコープ管理

引数 `$ARGUMENTS` を確認し、分析対象を決定してください：

- **引数なし** → `src/` 全体を対象に分析。※ただし、ファイル数が膨大でコンテキスト上限を超える可能性がある場合は、一度に全読み込みせず「ディレクトリごとに分割して実行するか、特定のドメインに絞るか」をユーザーに提案して確認を取ること。
- **ディレクトリ・ファイルパス** → 指定された範囲を対象に分析。
- **機能名・キーワード** → 自身のファイル検索ツール（Glob/Grep等）を実行して関連ファイルを特定してから分析。

### Step 2: コードの読み込みと深い分析

対象コードを読み込み、以下の観点で問題を洗い出してください。**表面的なコードの書き方（Lintレベル）のレビューではなく、設計の根拠・アーキテクチャパターンの妥当性まで掘り下げること。**

※対象プロジェクトの技術スタック（言語・フレームワーク）に合わせて、以下の観点を適宜読み替えること。

**設計の正確性**
- DDD等の設計概念（エンティティ・値オブジェクト・ドメインサービス等）の正しい適用
- IDの安定性（`randomUUID()` のレンダリング時生成などの誤用がないか）
- 責務の分離が実際に機能しているか（Fat Controller/Modelになっていないか）

**型安全性（TypeScript等）**
- `as` キャストによる型チェックの強制回避
- `never` 型を使った網羅性チェック（Exhaustiveness check）の欠如
- `unknown`/`any` が境界（APIレスポンス等）で適切に検証・パースされているか

**UI/コンポーネントパターン（React等）**
- `useSyncExternalStore` やHooksの参照安定性・依存配列の妥当性
- モジュールレベルのミュータブル状態の悪用
- `key` 属性の安定性と再レンダリングの最適化

**非同期処理・状態管理**
- `Promise.allSettled` の失敗が適切にハンドリングされ、無音エラーになっていないか
- AbortControllerによるリクエストキャンセルの考慮
- ライフサイクルやレースコンディションへの対応

**アーキテクチャ整合性とオブザーバビリティ**
- 既存のアーキテクチャルール（例: `.claude/rules/` 内の定義）との乖離
- DI（依存性の注入）が実際にテスタビリティの向上に寄与しているか
- エラーの握り潰し（グレースフルデグラデーションと単なる握り潰しを明確に区別する）
- 本番障害時に原因を追跡できる適切なログ・コンテキストが残されているか

**知識ギャップの特定（チームの学習領域）**
- 見つかった問題の傾向から、開発チーム全体で理解が不足していると思われる技術的・設計的テーマ（知識ギャップ）を推測すること。

### Step 3: 問題のトリアージ

見つかった問題を以下の基準で分類してください：

- **重大**: 設計の根本が間違っている。セキュリティリスクがある。将来の機能開発や保守を著しく阻害する。
- **中**: 特定条件でバグとして顕在化する。技術的負債として蓄積し、後々のリファクタリングコストが高くなる。
- **低**: 可読性・デバッグ容易性・拡張性の改善。局所的なパフォーマンス問題。

**※問題が0件の場合**: ファイルは作成せず、ユーザーに「設計上の問題は見つかりませんでした」と報告してプロセスを終了してください。

### Step 4: 分析結果をファイルに保存

システムの日付を使用して、`.analysis/YYYY-MM-DD-<対象名>-review.md` に分析結果をファイルとして出力・保存してください。

**ファイルの構成（マークダウン）**:
